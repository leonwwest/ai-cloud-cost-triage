Data Governance · Unauthorized Data Exposure

AI-Data-Governance Incident Runbook

Mini-portfolio piece · single-page runbook for data-access incidents via internal AI agents
Scenario

An internal AI agent answers questions using data the asking user should not have access to — e.g., another tenant's/customer's data, HBI/PII documents, or another department's data. This is a data-governance incident: the agent exposed data because the retriever/connector did not enforce the user's authorized scope (missing row-level security, over-broad service principal, missing delegation, classification-unaware retrieval).

The 8 Steps in Detail

1

Symptom — Confirm & classify exposure

What to check

  • User reports the answer contains foreign data? Or DLP/classification alert on agent output?
  • Data classification of the exposed item: Public / Internal / HBI / PII? Cross-tenant?
  • Reproducible? Pin the example query + answer (golden case).

Concrete action

  • Preserve the exposed content (screenshot/answer JSON) before caches are cleared.
  • Check classification via the data catalog (Purview/Dataplex/Macie).
  • First triage: single case or systemic (multiple users/sources)?
Stop if: PII/HBI or cross-tenant confirmed → pull in Security + DPO/Compliance immediately, don't debug on your own.
2

User/Group — who shouldn't have what

What to check

  • Which user/group/tenant saw the exposed data?
  • Should the user have no access to the source at all, or only a subset (row-level)?
  • A legitimate question wrongly answered from a foreign scope?

Concrete action

  • Read the user's identity, group/role assignments and tenant membership.
  • Compare against the entitlement matrix: which sources/rows/docs are allowed for this identity.
  • Was the exposed data already shared/exported? (answer history, copy/download logs.)
Stop if: multiple tenants/groups affected → switch from „single case" to „systemic", enumerate all affected.
3

IAM/RBAC — delegation vs. shared principal

What to check

  • Under which identity does the agent retrieve data? Delegated (on-behalf-of the user) or shared service principal?
  • Effective permissions of the principal per source (Storage Blob-Reader vs. Contributor, Lake Formation permissions, BigQuery roles).
  • Row-level security / ABAC policies present and active? Group membership in token claims?

Concrete action

  • Principal scheme: shared principal → root-cause candidate (sees all data).
  • Recent RBAC/policy changes on principal and sources (audit log).
  • Verify the RLS policy actually evaluates user claims/tenant ID.
Stop if: agent uses a shared principal without RLS → root cause; any question can pull foreign data.
4

Source/Connector — find the scope leak

What to check

  • Which connector/source returned the exposed rows/docs?
  • Connector filters: tenant ID, workspace, row-level, classification predicate?
  • Embedding space: are foreign and allowed documents in the same index without a filter constraint?

Concrete action

  • Trace the connector config and applied filters per request.
  • Check index partitioning: is a foreign tenant in the same vector index?
  • Compare source classification (HBI/PII) vs. allowed user scope.
Stop if: connector has no row/tenant filter → fix here (filter or index split), not just the symptom.
5

Logs/Query-Audit — what really hit the prompt

What to check

  • Query audit: executed query, filters, returned rows/docs, by identity.
  • Which contents were injected into the prompt? Reconstruct prompt context.
  • Audit coverage: is every retrieval call logged (correlation/trace ID end-to-end)?

Concrete action

  • Pull audit logs (Purview / CloudTrail Lake / BigQuery INFORMATION_SCHEMA.JOBS).
  • Per exposed answer: trace which rows/chunks were in context.
  • Simulate entitlement at runtime: should RLS have blocked these rows?
Stop if: audit gaps (retrieval not logged) → its own weakness; forensics limited, add to the alert rule (step 8).
6

Cost/Tokens — over-retrieval as secondary signal

What to check

  • prompt_tokens of affected requests vs. baseline: over-retrieval inflates context.
  • Scope creep: connector fetches more rows than needed (missing top-k/filter cap).
  • Cost anomaly correlated with exposure requests?

Concrete action

  • Compare P50/P95 of prompt_tokens for affected user/source.
  • Check retrieval top-k and fetch limits.
  • Cross cost/token signal with governance signal (often the same requests).
Stop if: tokens/req P95 far above baseline → over-retrieval amplifies exposure risk; cap top-k in the immediate action.
7

Immediate Action — plug scope, purge exposure

What to check

  • Is the cause narrowed (connector/principal/RLS)? Otherwise pause the agent.
  • Has exposed data already been shared (answer history, exports)?
  • Reporting duties (PII/HBI → DPO/Compliance, possibly regulator)?

Concrete action

  • Disable the affected connector or hard-set it to the authorized scope.
  • Lock the shared principal → switch to delegated identity (on-behalf-of).
  • Delete answer history/caches of exposed answers; inform affected users + data owner.
  • For PII/HBI: trigger security IR, preserve evidence, notify DSO/DPO.
Stop if: PII/HBI or cross-tenant exposed → let security IR take over; don't debug-delete on your own.
8

Alert Rule — standing detection + routing

What to check

  • How should this incident have been detected automatically?
  • Alert routing: who is notified (SecOps/data owner/DPO)? Runbook linked?
  • Realistic false-positive rate? Distinguish from a mere „long answer".

Concrete action

  • DLP/classification alert on agent output (HBI/PII pattern match).
  • Query-audit alert: retrieval rows outside user entitlement → row-level-violation-count > 0.
  • RAG-grounding alert: prompt context contains sources outside the allowed scope.
  • Route to PagerDuty/SecOps; ownership + runbook link in the alert.
Stop if: alert only goes to an email list → insufficient; couple to paging/SecOps.

Cloud CLI & Query Snippets

Copy-paste commands per cloud for governance forensics. Raw files in snippets/ in the repo.

# Effective permissions of the agent principal per source
az role assignment list --assignee <agent-object-id> --all \
  --query "[].{role:roleDefinitionName, scope:scope}" -o table

# Last RBAC/policy changes (Activity Log)
az monitor activity-log list --caller <agent-object-id> \
  --resource-type "Microsoft.Authorization/roleAssignments"

# Purview: sensitivity classification of a source
az purview classification-rule list --endpoint <purview-account> --wait 0

# Storage: row-level/ACL scope of a container
az storage container policy list --account-name <storage> --name <container>
az storage container show-permission --account-name <storage> --name <container>

# CloudTrail Lake: reconstruct retrieval audit by identity
aws cloudtrail start-query --query-statement '
  SELECT eventTime, userIdentity.arn, eventName, requestParameters,
         resources[1].ARN
  FROM <EVENT_DATA_STORE>
  WHERE userIdentity.sessionIssuer.arn LIKE ''%AgentRole%''
    AND eventTime > ''2026-07-01'''

# Lake Formation: row-/column-level permissions of the agent
aws lakeformation list-permissions --principal <agent-role-arn>

# Macie: find PII in the exposed source (classification)
aws macie2 describe-classification-job --job-id <job-id>

# IAM: agent role + session policy
aws iam get-role --role-name AgentServiceRole
aws iam list-role-policies --role-name AgentServiceRole

# BigQuery: audit per user/job (which rows were pulled)
bq query --use_legacy_sql=false '
  SELECT job_user, query, total_slot_ms, statement_type
  FROM `region-eu`.INFORMATION_SCHEMA.JOBS_BY_PROJECT
  WHERE creation_time >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 DAY)
    AND job_user LIKE "%agent%"'

# Dataplex: data classification / policy tags of the source
gcloud dataplex entries describe --entry=projects/<p>/locations/eu/entryTypes/<t>

# IAM: effective permissions of the agent service account
gcloud asset analyze-iam-policy --full-resource-name=<RESOURCE> \
  --permissions="bigquery.datasets.get" --format=json

// Retrieval audit: rows/docs per user + connector + scope
AIAgentRetrieval
| where timestamp > ago(24h)
| summarize rows=count(), docs=count_distinct(docId) by userId, connector, scope
| order by rows desc

// Reconstruct prompt context of an exposed answer
AIAgentPrompt
| where operation_Id == "<trace-id>"
| extend ctx = parse_json(promptContext)
| mv-apply c = ctx on (project source=c.source, docId=c.docId, classification=c.classification)
| join kind=leftantisemi (UserEntitlements | where allowed == false) on $left.docId == $right.docId

// Alert basis: row-level violations per hour
AIAgentRetrieval
| extend violation = iff(scope !in (entitlement), 1, 0)
| summarize violations = sum(violation) by bin(1h), userId
| where violations > 0

Prevention & Guardrails

Reactive is too late — these guardrails prevent over-exposure systemically.

  • Delegated identity (on-behalf-of) — agent retrieves data as the user, not as a shared principal; authorization enforced per user.
  • Row-level security enforced — sources filter by tenant/group/claims; policy forbids shared-principal access.
  • Index partitioning by tenant — foreign data not in the same vector index; embedding scope = entitlement scope.
  • Classification-aware retrieval — HBI/PII sources only for entitled scope; matched against data-catalog labels.
  • Top-k & fetch limits — hard-capped retrieval; prevents over-retrieval and reduces exposure surface.
  • Prompt-grounding gate — before the model call, verify all context sources are in the allowed scope (CI/runtime guard).
  • DLP on agent output — PII/HBI pattern match on answers; on hit, block + alert.
  • Query-audit coverage SLO — 100% of retrieval calls logged with user identity; gaps become tickets.

Worked Post-Mortem — Case #077

A walkthrough showing how the flow uncovers the root cause.

IncidentINC-2026-077 · AI agent „HelpDesk v3" · P1 · Duration 4h 10min · Classification: HBI / Cross-Tenant
SymptomSupport agent (Tenant A) asks „what are our SLA numbers?" — the answer cites Tenant B customer data incl. customer name and contract values.
Impact1 user saw HBI data of another tenant; no onward sharing, no PII (contract data); potential compliance breach reported.
Root CauseRAG connector used a shared service principal with Storage Blob Data Reader over the whole container across all tenants. No row-level security; embeddings of all tenants in a shared vector index. The Tenant-A question embedding matched Tenant-B documents by chance.
Detection GapNo DLP on agent output; query audit ran but no alert on scope ≠ entitlement; detection only via user report.
Fix (1st hour)Agent paused; connector disabled; shared principal locked; exposed answer deleted from history/cache; Tenant-B owner + DPO informed.
Lasting FixSwitch to on-behalf-of delegation; enable RLS per tenant; partition vector index by tenant; DLP on output; query-audit alert on row-level violations; PagerDuty to SecOps.
Lessons(1) A shared principal is a governance weakness, not just convenience. (2) A shared vector index = shared exposure. (3) User reports are not a detection concept.

Triage Timeline

  1. T+0

    Symptom reported

    User reports foreign data in the answer. Exposure preserved; classification: HBI / cross-tenant → Security + DPO informed immediately.

  2. T+15min

    User/Group

    One user, Tenant A. Allowed scope: Tenant A only. Legitimate question, wrong source.

  3. T+35min

    IAM/RBAC

    Agent uses a shared service principal, no on-behalf-of. Effective permissions: reader over the whole container → root-cause candidate.

  4. T+50min

    Source/Connector

    Connector has no tenant filter; vector index contains all tenants. Tenant-A question embedding matched Tenant-B docs.

  5. T+1h 10min

    Logs/Query-Audit — cause proven

    Audit shows retrieval of Tenant-B docs for a Tenant-A user. RLS should have blocked it, but was never configured.

  6. T+1h 25min

    Cost/Tokens

    Affected requests P95 = 22k tokens (baseline 4k) — over-retrieval over the whole index amplified exposure.

  7. T+1h 40min

    Immediate action

    Agent paused, connector off, principal locked, exposure deleted, DPO + Tenant-B owner informed. Security IR takes over forensics.

  8. T+4h 10min

    Alert rule & close

    Query-audit alert on scope ≠ entitlement + DLP on output implemented; lasting fixes logged as action items with owner & ETA.

Decision Tree (Short Form)

Agent answers with foreign data?
├── PII/HBI or cross-tenant? → Security + DPO, let IR take over (1)
├── one user or many?
│   ├── many → systemic: shared principal / index partition (3,4)
│   └── one → connector filter / single entitlement (2,4)
├── agent uses shared principal? → switch to delegation (3,7)
├── RLS/filter missing? → hard-scope the connector (4,7)
├── shared vector index? → partition by tenant (4)
└── audit gaps? → add to the alert rule (5,8)

Quick-Fixes (first hour)

  1. Pause agent / disable connector — stop further over-exposure until scope is clear.
  2. Lock shared principal — prepare the move to on-behalf-of delegation.
  3. Delete exposed answers — from answer history & caches; check onward sharing.
  4. Cap top-k — hard-limit retrieval until filters/RLS are in place.
  5. Inform Security/DPO — on PII/HBI/cross-tenant immediately; don't debug-delete on your own.
