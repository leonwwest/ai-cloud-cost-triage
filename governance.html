Data Governance · Unauthorized Data Exposure

AI-Data-Governance Incident Runbook

Mini-Portfolio-Stück · einseitiger Runbook für Datenzugriffs-Incidents durch interne AI-Agenten
Szenario

Ein interner AI-Agent beantwortet Fragen mit Daten, auf die der fragende User eigentlich keinen Zugriff haben sollte — z. B. Daten eines anderen Tenants/Kunden, HBI-/PII-Dokumente oder fremde Abteilungsdaten. Das ist ein Daten-Governance-Vorfall: der Agent hat Daten exponiert, weil Retriever/Connector den berechtigten Scope des Users nicht durchgesetzt haben (fehlendes Row-Level-Security, überbreiter Service-Principal, fehlende Delegation, klassifikationsunbewusstes Retrieval).

Die 8 Schritte im Detail

1

Symptom — Exposition bestätigen & klassifizieren

Was prüfen

  • Meldung eines Users, dass die Antwort fremde Daten enthält? Oder DLP-/Klassifizierungs-Alert auf Agent-Output?
  • Daten-Klassifizierung des Exponats: Public / Internal / HBI / PII? Cross-Tenant?
  • Reproduzierbar? Beispiel-Query + Antwort fixieren (Golden-Case).

Konkrete Aktion

  • Exponierten Inhalt sichern (Screenshot/Antwort-JSON) bevor Caches gelöscht werden.
  • Klassifizierung über Data-Catalog (Purview/Dataplex/Macie) prüfen.
  • Erste Einschätzung: Einzelfall oder systemisch (mehrere User/Quellen)?
Stopp, falls: PII/HBI oder Cross-Tenant bestätigt → Security + DPO/Compliance sofort rein, nicht selbst weiter debuggen.
2

Betroffener User/Gruppe — wer darf was nicht

Was prüfen

  • Welcher User/Gruppe/Tenant hat die exponierten Daten gesehen?
  • Sollte der User auf die Quelle grundsätzlich keinen Zugriff haben, oder nur auf einen Subset (Row-Level)?
  • Eigentlich legitime Frage, die fälschlich aus fremdem Scope beantwortet wurde?

Konkrete Aktion

  • Identität, Gruppen/Role-Assignments und Tenant-Zugehörigkeit des Users auslesen.
  • Abgleich: welche Quellen/Rows/Docs für diese Identität erlaubt sind (Entitlement-Matrix).
  • Wurden exponierte Daten bereits geteilt/exportiert? (Antwort-Historie, Copy/Download-Logs prüfen.)
Stopp, falls: mehrere Tenants/Gruppen betroffen → von „Einzelfall" auf „systemisch" umstellen, alle Betroffenen erfassen.
3

IAM/RBAC prüfen — Delegation vs. Shared-Principal

Was prüfen

  • Mit welcher Identität ruft der Agent Daten ab? Delegiert (On-Behalf-Of des Users) oder Shared Service-Principal?
  • Effective Permissions des Principals pro Quelle (Storage-Blob-Reader vs. Contributor, Lake Formation permissions, BigQuery roles).
  • Row-Level-Security / ABAC-Policies vorhanden und aktiv? Gruppenzugehörigkeit im Token claims?

Konkrete Aktion

  • Principal-Schema prüfen: bei shared principal → Root-Cause-Kandidat (sieht alle Daten).
  • Letzte RBAC-/Policy-Änderungen an Principal und Quellen (Audit-Log).
  • Verifizieren, ob RLS-Policy die User-Claims/Tenant-ID tatsächlich auswertet.
Stopp, falls: Agent nutzt Shared-Principal ohne RLS → übergeordnete Ursache; jede Frage kann fremde Daten ziehen.
4

Datenquelle/Connector prüfen — Scope-Verletzung finden

Was prüfen

  • Welcher Connector/Quelle hat die exponierten Rows/Docs zurückgeliefert?
  • Connector-Filter: Tenant-ID, Workspace, Row-Level, Klassifizierungs-Prädikat vorhanden?
  • Embedding-Space: liegen fremde und erlaubte Dokumente im selben Index ohne Filter-Constraint?

Konkrete Aktion

  • Connector-Konfiguration + angewandte Filter pro Request nachvollziehen.
  • Index-Partitionierung prüfen: ist fremder Tenant im selben Vektor-Index?
  • Klassifizierung der Quelle (HBI/PII) vs. erlaubte User-Scope abgleichen.
Stopp, falls: Connector hat keinen Row-/Tenant-Filter → Fix hier (Filter oder Index-Trennung), nicht nur Symptom.
5

Logs/Query-Audit — was landete wirklich im Prompt

Was prüfen

  • Query-Audit: ausgeführte Query, Filter, zurückgegebene Rows/Docs, identitätsbezogen.
  • Welche Inhalte wurden ins Prompt injiziert? Prompt-Kontext rekonstruieren.
  • Audit-Coverage: ist jeder Retrieval-Call geloggt (Correlation-/Trace-ID durchgängig)?

Konkrete Aktion

  • Audit-Logs ziehen (Purview / CloudTrail Lake / BigQuery INFORMATION_SCHEMA.JOBS).
  • Pro exponierter Antwort: nachvollziehen, welche Rows/Chunks im Kontext standen.
  • Entitlement zur Laufzeit simulieren: hätte RLS die Rows blockieren müssen?
Stopp, falls: Audit-Lücken (Retrieval nicht geloggt) → eigene Schwachstelle; Forensik eingeschränkt, in Alert-Regel (Schritt 8) aufnehmen.
6

Kosten/Tokenverbrauch — Over-Retrieval als Sekundärsignal

Was prüfen

  • prompt_tokens der betroffenen Requests vs. Basislinie: Over-Retrieval bläht den Kontext.
  • Scope-Creep: Connector holt mehr Rows als nötig (fehlende Top-K-/Filter-Begrenzung).
  • Kostenanomalie korreliert mit Expositions-Requests?

Konkrete Aktion

  • P50/P95 von prompt_tokens für betroffene User/Quelle vergleichen.
  • Top-K und Fetch-Grenzen des Retrievals prüfen.
  • Kosten-/Token-Signal mit Governance-Signal kreuzen (oft gleiche Requests).
Stopp, falls: Tokens/req P95 weit über Basis → Over-Retrieval verstärkt Expositions-Risiko; in Sofortmaßnahme Top-K deckeln.
7

Sofortmaßnahme — Scope stopfen, Exponate löschen

Was prüfen

  • Ist die Ursache eingegrenzt (Connector/Principal/RLS)? Sonst Agent pausieren.
  • Exponierte Daten bereits weitergegeben (Antwort-Historie, Export)?
  • Meldungspflichten (PII/HBI → DPO/Compliance, ggf. Aufsichtsbehörde)?

Konkrete Aktion

  • Betroffenen Connector deaktivieren oder auf authorisierten Scope hart setzen.
  • Shared-Principal sperren → auf delegierte Identität (On-Behalf-Of) umstellen.
  • Antwort-Historie/Caches der exponierten Antworten löschen; betroffene User + Daten-Owner informieren.
  • Bei PII/HBI: Security-IR anstoßen, Beweise sichern, Meldung an DSO/DPO.
Stopp, falls: PII/HBI oder Cross-Tenant exponiert → Security-IR übernehmen lassen; nicht selbst löschend debuggen.
8

Dauerhafte Alert-Regel — Erkennung + Routing

Was prüfen

  • Wie hätte der Vorfall automatisch erkannt werden sollen?
  • Alert-Routing: wer wird benachrichtigt (SecOps/Data-Owner/DPO)? Runbook verlinkt?
  • False-Positive-Rate realistisch? Alert gegen reine „große Antwort" abgrenzen.

Konkrete Aktion

  • DLP-/Klassifizierungs-Alert auf Agent-Output (HBI/PII-Pattern-Match).
  • Query-Audit-Alert: Retrieval-Rows außerhalb User-Entitlement → row-level-violation-count > 0.
  • RAG-Grounding-Alert: Prompt-Kontext enthält Quellen außerhalb erlaubtem Scope.
  • Routing an PagerDuty/SecOps; Ownership + Runbook-Link im Alert.
Stopp, falls: Alert geht nur an E-Mail-Liste → unzureichend; an Paging/SecOps koppeln.

Cloud-CLI & Query-Snippets

Copy-Paste-Befehle je Cloud für Governance-Forensik. Rohdateien unter snippets/ im Repo.

# Effective permissions des Agent-Principals pro Quelle
az role assignment list --assignee <agent-object-id> --all \
  --query "[].{role:roleDefinitionName, scope:scope}" -o table

# Letzte RBAC-/Policy-Änderungen (Audit-Log)
az monitor activity-log list --caller <agent-object-id> \
  --resource-type "Microsoft.Authorization/roleAssignments"

# Purview: Sensitivitäts-Klassifizierung einer Quelle prüfen
az purview classification-rule list --endpoint <purview-account> --wait 0

# Storage: Row-Level/ACL-Scope eines Containers prüfen
az storage container policy list --account-name <storage> --name <container>
az storage container show-permission --account-name <storage> --name <container>

# CloudTrail Lake: Retrieval-Audit pro Identität rekonstruieren
aws cloudtrail start-query --query-statement '
  SELECT eventTime, userIdentity.arn, eventName, requestParameters,
         resources[1].ARN
  FROM edf ... WHERE userIdentity.sessionIssuer.arn LIKE ''%AgentRole%''
  AND eventTime > ''2026-07-01'''

# Lake Formation: Row-Level-/Column-Level-Permissions des Agent
aws lakeformation list-permissions --principal <agent-role-arn>

# Macie: PII in der exponierten Quelle finden (Klassifizierung)
aws macie2 describe-classification-job --job-id <job-id>

# IAM: Agent-Role + Session-Policy prüfen
aws iam get-role --role-name AgentServiceRole
aws iam list-role-policies --role-name AgentServiceRole

# BigQuery: Audit pro User/Job (welche Rows wurden gezogen)
bq query --use_legacy_sql=false '
  SELECT job_user, query, total_slot_ms, statement_type
  FROM `region-eu`.INFORMATION_SCHEMA.JOBS_BY_PROJECT
  WHERE creation_time >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 DAY)
    AND job_user LIKE "%agent%"'

# Dataplex: Daten-Klassifizierung / Policy-Tags der Quelle
gcloud dataplex entries describe --entry=projects/<p>/locations/eu/entryTypes/<t>

# IAM: Effective permissions des Agent-Service-Accounts
gcloud asset analyze-iam-policy --full-resource-name=<RESOURCE> \
  --permissions="bigquery.datasets.get" --format=json

// Retrieval-Audit: Rows/Docs pro User + Connector + Scope
AIAgentRetrieval
| where timestamp > ago(24h)
| summarize rows=count(), docs=count_distinct(docId) by userId, connector, scope
| order by rows desc

// Prompt-Kontext einer exponierten Antwort rekonstruieren
AIAgentPrompt
| where operation_Id == "<trace-id>"
| extend ctx = parse_json(promptContext)
| mv-apply c = ctx on (project source=c.source, docId=c.docId, classification=c.classification)
| join kind=leftantisemi (UserEntitlements | where allowed == false) on $left.docId == $right.docId

// Alert-Basis: Row-Level-Verletzungen pro Stunde
AIAgentRetrieval
| extend violation = iff(scope !in (entitlement), 1, 0)
| summarize violations = sum(violation) by bin(1h), userId
| where violations > 0

Prevention & Guardrails

Reaktiv ist zu spät — diese Guardrails verhindern Over-Exposure systemisch.

  • Delegierte Identität (On-Behalf-Of) — Agent ruft Daten als User ab, nicht als Shared-Principal; Autorisierung wird pro User durchgesetzt.
  • Row-Level-Security erzwungen — Datenquellen filtern nach Tenant/Group/Claims; Policy verhindert Shared-Principal-Zugriff.
  • Index-Partitionierung nach Tenant — fremde Daten nicht im selben Vektor-Index; Embedding-Scope = Berechtigungsscope.
  • Klassifizierungs-bewusstes Retrieval — HBI/PII-Quellen nur für berechtigte Scope; Matching gegen Data-Catalog-Labels.
  • Top-K- & Fetch-Limits — hart begrenztes Retrieval; verhindert Over-Retrieval und reduziert Expositionsfläche.
  • Prompt-Grounding-Gate — vor Modellaufruf prüfen, dass alle Kontext-Quellen im erlaubten Scope (CI/Runtime-Guard).
  • DLP auf Agent-Output — PII/HBI-Pattern-Match auf Antworten; bei Treffer blocken + alerten.
  • Query-Audit-Coverage-SLO — 100 % der Retrieval-Calls geloggt mit User-Identität; Lücken werden Tickets.

Worked Post-Mortem — Case #077

Ein durchgespielter Governance-Vorfall, der zeigt, wie der Fluss die Ursache aufdeckt.

IncidentINC-2026-077 · AI-Agent „HelpDesk v3" · P1 · Dauer 4h 10min · Klassifikation: HBI / Cross-Tenant
SymptomSupport-Mitarbeiter (Tenant A) fragt „Wie sind unsere SLA-Zahlen?" — Antwort zitiert Kundendaten von Tenant B inkl. Kundenname & Vertragssummen.
Impact1 User sah HBI-Daten eines anderen Tenants; keine Weitergabe, keine PII (Vertragsdaten); potenzieller Compliance-Verstoß gemeldet.
Root CauseRAG-Connector nutzte Shared Service-Principal mit Storage Blob Data Reader auf dem gesamten Container über alle Tenants. Keine Row-Level-Security; Embeddings aller Tenants im gemeinsamen Vektor-Index. Frage-A Embedding matchte zufällig Dokumente von Tenant B.
Detection-GapKeine DLP auf Agent-Output; Query-Audit lief, aber kein Alert auf scope ≠ entitlement; Erkennung erst durch User-Meldung.
Fix (1. Stunde)Agent pausiert; Connector deaktiviert; Shared-Principal gesperrt; exponierte Antwort aus Historie/Cache gelöscht; Tenant B-Owner + DPO informiert.
Dauer-FixUmstellung auf On-Behalf-Of-Delegation; RLS pro Tenant aktiviert; Vektor-Index nach Tenant partitioniert; DLP auf Output; Query-Audit-Alert auf Row-Level-Verletzungen; PagerDuty an SecOps.
Lessons(1) Shared-Principal ist eine Governance-Schwachstelle, nicht nur eine Convenience. (2) Gemeinsamer Vektor-Index = gemeinsame Exposition. (3) User-Reports sind kein Detection-Konzept.

Triage-Verlauf (Timeline)

  1. T+0

    Symptom gemeldet

    User meldet fremde Daten in der Antwort. Exponat gesichert; Klassifizierung: HBI / Cross-Tenant → Security + DPO sofort informiert.

  2. T+15min

    User/Gruppe

    Ein User, Tenant A. Erlaubter Scope: nur Tenant A. Eigentlich legitime Frage, falsche Quelle.

  3. T+35min

    IAM/RBAC

    Agent nutzt Shared Service-Principal, kein On-Behalf-Of. Effective Permissions: Reader auf dem Gesamt-Container → Root-Cause-Kandidat.

  4. T+50min

    Datenquelle/Connector

    Connector ohne Tenant-Filter; Vektor-Index enthält alle Tenants. Frage-A-Embedding matchte Tenant-B-Dokumente.

  5. T+1h 10min

    Logs/Query-Audit — Ursache bewiesen

    Audit zeigt Retrieval von Tenant-B-Docs für Tenant-A-User. RLS hätte das blocken müssen, war aber gar nicht konfiguriert.

  6. T+1h 25min

    Kosten/Token

    Betroffene Requests P95 = 22k Tokens (Basis 4k) — Over-Retrieval über den Gesamt-Index hat Exposition verstärkt.

  7. T+1h 40min

    Sofortmaßnahme

    Agent pausiert, Connector aus, Principal gesperrt, Exponat gelöscht, DPO + Tenant-B-Owner informiert. Security-IR übernimmt Forensik.

  8. T+4h 10min

    Alert-Regel & Close

    Query-Audit-Alert auf scope ≠ entitlement + DLP auf Output implementiert; Dauer-Fixes als Action-Items mit Owner & ETA.

Entscheidungsbaum (Kurzform)

Agent antwortet mit fremden Daten?
├── PII/HBI oder Cross-Tenant? → Security + DPO, IR übernehmen lassen (1)
├── ein User oder viele?
│   ├── viele → systemisch: Shared-Principal / Index-Partition prüfen (3,4)
│   └── einer → Connector-Filter / einzelne Entitlement prüfen (2,4)
├── Agent nutzt Shared-Principal? → auf Delegation umstellen (3,7)
├── RLS/Filter fehlt? → Connector hart scope-en (4,7)
├── gemeinsamer Vektor-Index? → nach Tenant partitionieren (4)
└── Audit-Lücken? → in Alert-Regel aufnehmen (5,8)

Quick-Fixes (erste Stunde)

  1. Agent pausieren / Connector deaktivieren — bis Scope geklärt ist, weiteren Over-Exposure stoppen.
  2. Shared-Principal sperren — Übergang auf On-Behalf-Of-Delegation vorbereiten.
  3. Exponierte Antworten löschen — aus Antwort-Historie & Caches; Weitergabe prüfen.
  4. Top-K deckeln — Retrieval hart begrenzen, bis Filter/RLS stehen.
  5. Security/DPO informieren — bei PII/HBI/Cross-Tenant sofort, nicht selbst löschend debuggen.
