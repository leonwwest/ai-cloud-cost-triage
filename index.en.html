AI-Cloud Cost & Access Triage

Mini-portfolio piece · single-page triage runbook for internal AI agents
Scenario

A team uses an internal AI agent. Costs spike suddenly and noticeably. At the same time, responses reference wrong / unauthorized data sources. Both symptoms appearing together = typically one shared root cause (e.g., misconfigured data-source connector, wrong API key, leaking prompt context, rolled-out RBAC change).

The 9 Steps in Detail

1

Symptom — Quantify the observation

What to check

  • Cost delta: daily/hourly average vs. 7-day baseline (absolute and per call).
  • Answer quality: which responses reference wrong data? Reproducible?
  • Time correlation: when did both symptoms start? Same timestamp = shared cause.

Concrete action

  • Metrics plot for 14 days: cost/day, tokens/req, error_rate, mismatch tickets.
  • Pin sample queries referencing the wrong data source (golden case).
Stop if: cost and quality symptoms start at the same timestamp → treat as one incident.
2

User/Team — Affected & culprit

What to check

  • Which teams/users report wrong answers? Concentrated or spread?
  • Which agent service / agent_version produces the cost?
  • Recent changes: feature rollout, new data connector, prompt change?

Concrete action

  • Group calls by tenant/team/user_id and agent_version.
  • Diff the last deployment (prompt template, tool/function definitions, retrieval sources).
Stop if: a single service account or agent_version dominates → narrow down there.
3

Model/API-Key — Which model, which key

What to check

  • Costlier model deployed? (gpt-4o vs. 4o-mini, claude-opus vs. sonnet).
  • API key origin: production vs. test/shared key? Rotation date?
  • Routing: fallback to costlier model on error? RAG enabled?

Concrete action

  • Per key: requests, tokens_in/out, cost for last 24h from billing/usage API.
  • Check key usage for anomalies (unexplained usage = possible leak).
Stop if: key appears in code/logs/public repos → rotate immediately, treat as possible abuse.
4

Token Cost — Where tokens originate

What to check

  • Prompt tokens vs. completion tokens. High prompt tokens = context/retrieval blowup.
  • System prompt size per request; bundled docs/tool definitions.
  • Loops/retries: does the agent call the model multiple times per task?

Concrete action

  • Distribution of prompt_tokens per request (P50/P95/P99). P95 » P50 = outliers.
  • Inspect top-10 requests by token usage → often 1–2 main culprits.
  • Check prompt caching for static system prompts.
Stop if: prompt-tokens P95 suddenly 5–10× higher → suspect data-source context / loop.
5

IAM/RBAC — Are accesses authorized

What to check

  • Agent identity: managed identity / service principal / user?
  • Granted vs. needed permissions (least privilege): Blob-Reader vs. Contributor.
  • Is there an allow-list of tools/data sources?

Concrete action

  • Read effective permissions of the agent principal per data source.
  • Recent RBAC changes on principal / data sources (audit log).
  • Access to sources outside the intended scope? → explains „wrong data".
Stop if: principal is Contributor/Owner at broad scope → over-privilege is likely cause.
6

Private Endpoint/Network — Who may talk to whom

What to check

  • Does the agent only talk to allow-listed endpoints? (Private Endpoint, VNet/Subnet, egress rules).
  • New data source connected via public IP / non-private endpoint?
  • DNS: does the name resolve to the private endpoint or publicly?

Concrete action

  • Check egress/NSG flow logs for last 24h → unexpected target IPs/domains.
  • Verify every used source is reachable via Private/Service Endpoint.
  • RAG connector using a public endpoint? Data may come from another tenant/workspace.
Stop if: agent reaches sources outside the network perimeter → cause for „wrong data" + uncontrolled fetches.
7

Logs/Traces — What actually happened

What to check

  • End-to-end trace: User → Agent → Retrieval → Data source → Model → Response.
  • Were multiple sources queried? What ended up in the prompt context?
  • Tool/function call logs: which tools, which arguments?

Concrete action

  • Pull distributed tracing (trace ID) for sample cases.
  • Reconstruct prompt context of a wrong answer → injected sources.
  • Follow correlation ID across all hops; missing spans = observability gaps.
Stop if: trace shows retrieval from unexpected source → proves the cause; fix in step 5/6.
8

Budget Alert — Is early warning working

What to check

  • Was there a budget/anomaly alert? When did it fire?
  • Was it ignored / not routed (email silo, missing paging)?
  • Are thresholds realistic? Anomaly detection vs. fixed threshold.

Concrete action

  • Check alert history for 14 days; latency between event and alert.
  • Add alerting on tokens/req P95 and cost/hour (not just cost/day).
  • Clarify ownership; add runbook link to the alert.
Stop if: alert fired but sat 6+ h unattended → post-incident, review alerting/on-call process too.
9

Escalation — Who needs to be involved

What to check

  • Security suspicion? (leaked key, foreign data source, unauthorized egress).
  • Cost damage so far (for stakeholder communication).
  • Who owns it: platform, security, data owner, product owner?

Concrete action

  • Security suspicion → immediately trigger IR, rotate key, temporarily lock principal.
  • Path: L1 (Platform/Dev) → L2 (Security) → L3 (Data Owner + Management).
  • Stakeholder update: symptom, impact, working hypothesis, next action, ETA.
Stop if: security indicators (unexplained key usage, cross-tenant) → do not debug further; hand to IR owner, preserve evidence.

Cloud CLI & Query Snippets

Copy-paste commands per cloud. Raw files in snippets/ in the repo.

# Cost & usage per API key / model (Azure Cost Management)
az consumption usage list --top 50 \
  --query "[?contains(instanceName,'ai-agent')].{name:instanceName, cost:pretaxCost, date:usageStart}"

# Effective permissions of the agent principal on data sources
az role assignment list --assignee <agent-object-id> --all \
  --query "[].{role:roleDefinitionName, scope:scope}" -o table

# Last RBAC changes (Activity Log)
az monitor activity-log list --caller <agent-object-id> \
  --status Succeeded --resource-type "Microsoft.Authorization/roleAssignments"

# Private Endpoint connections of a Storage account
az storage account show --name <storage> --query "privateEndpointConnections"

# Cost Explorer: daily cost per service/tag (AWS)
aws ce get-cost-and-usage \
  --time-period Start=2026-06-01,End=2026-07-02 \
  --granularity DAILY --metrics "UnblendedCost" \
  --group-by Type=TAG,Key=agent_version Type=SERVICE

# CloudWatch Logs Insights: token distribution per request
aws logs start-query --log-group-name /ai/agent \
  --start-time $(date -d '7 days ago' +%s) --end-time $(date +%s) \
  --query-string 'filter event="llm_call"
  | stats pct(prompt_tokens,95) as p95, pct(prompt_tokens,99) as p99 by bin(1h)'

# IAM: list policies of the agent service role
aws iam list-attached-role-policies --role-name AgentServiceRole
aws iam get-role --role-name AgentServiceRole --query "Role.MaxSessionDuration"

# GCP Billing: daily cost per label (agent_version)
gcloud billing accounts list
gcloud alpha billing accounts describe <ACCOUNT> --format="value(displayName)"

# BigQuery: token cost export aggregated per request
bq query --use_legacy_sql=false '
  SELECT agent_version, APPROX_QUANTILES(prompt_tokens, 100)[OFFSET(95)] AS p95,
         SUM(cost) AS cost
  FROM `proj.ai.token_usage`
  WHERE _PARTITIONDATE >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 7 DAY)
  GROUP BY agent_version ORDER BY cost DESC'

# IAM effective permissions (Policy Analyzer)
gcloud asset analyze-iam-policy --full-resource-name=<RESOURCE> \
  --permissions="storage.objects.get" --format=json

// AppInsights/Log Analytics: token distribution + top consumers
let span = 24h;
AIAgentLLMCall
| where timestamp > ago(span)
| summarize req=count(), p50=percentile(prompt_tokens,50),
            p95=percentile(prompt_tokens,95), p99=percentile(prompt_tokens,99),
            cost=sum(todouble(cost)) by agent_version, dataSource
| where p95 > p50 * 3   // only requests with blowup
| order by cost desc

// Trace hops per trace ID (find observability gaps)
AppTraces
| where operation_Id == "<trace-id>"
| project timestamp, name, severityLevel, message
| order by timestamp asc

Budget Alert & Cost-Cap as Code

Budget + anomaly alert + action group as infrastructure-as-code. Files: infra/budget.tf (Terraform) & infra/budget.bicep.

# infra/budget.tf — Budget + Anomaly Alert + Action Group (Azure)
resource "azurerm_resource_group" "ai" {
  name     = "rg-ai-agent"
  location = var.location
}

resource "azurerm_monitor_action_group" "oncall" {
  name                = "ag-ai-oncall"
  resource_group_name = azurerm_resource_group.ai.name
  short_name          = "aiOnCall"
  email_receiver {
    name          = "platform"
    email_address = "platform-oncall@example.com"
  }
}

resource "azurerm_consumption_budget_resource_group" "agent" {
  name              = "budget-ai-agent"
  resource_group_id = azurerm_resource_group.ai.id
  amount                = 500                       # EUR / month
  time_grain            = "Monthly"
  time_period {
    start_date = "2026-07-01T00:00:00Z"
    end_date   = "2027-07-01T00:00:00Z"
  }
  filter {
    tag {
      name  = "agent_version"
      values = ["prod"]
    }
  }
  notification {
    enabled   = true
    threshold = 80                       # % of budget
    operator  = "GreaterThan"
    contact_groups = [azurerm_monitor_action_group.oncall.id]
  }
  notification {
    enabled   = true
    threshold = 100
    operator  = "GreaterThan"
    contact_groups = [azurerm_monitor_action_group.oncall.id]
  }
}

# Anomaly detection (scheduled query alert on cost/hour)
resource "azurerm_monitor_scheduled_query_rules_alert" "cost_spike" {
  name                = "alert-ai-cost-spike"
  resource_group_name = azurerm_resource_group.ai.name
  location            = azurerm_resource_group.ai.location
  severity            = 2
  frequency           = 5
  time_window         = 15
  data_source_id      = var.law_id
  query               = <<QUERY
AIAgentLLMCall | summarize cost = sum(todouble(cost)) by bin(1h)
| where cost > 20   # EUR/h threshold
QUERY
  trigger {
    operator  = "GreaterThan"
    threshold = 0
  }
  action {
    action_groups = [azurerm_monitor_action_group.oncall.id]
  }
}

// infra/budget.bicep — Budget + Alert (Azure)
param location string = resourceGroup().location
param budgetAmount int = 500
param oncallEmail string = 'platform-oncall@example.com'

resource actionGroup 'Microsoft.Insights/actionGroups@2023-01-01' = {
  name: 'ag-ai-oncall'
  location: 'global'
  properties: {
    groupShortName: 'aiOnCall'
    enabled: true
    emailReceivers: [{
      name: 'platform'
      emailAddress: oncallEmail
      useCommonAlertSchema: true
    }]
  }
}

resource budget 'Microsoft.Consumption/budgets@2023-04-01-preview' = {
  name: 'budget-ai-agent'
  properties: {
    amount: budgetAmount
    timeGrain: 'Monthly'
    timePeriod: {
      startDate: '2026-07-01T00:00:00Z'
      endDate:   '2027-07-01T00:00:00Z'
    }
    filter: {
      tags: { name: 'agent_version', values: ['prod'] }
    }
    notifications: {
      warn80: { enabled: true, operator: 'GreaterThan', threshold: 80, contactEmails: [oncallEmail] }
      over100: { enabled: true, operator: 'GreaterThan', threshold: 100, contactEmails: [oncallEmail] }
    }
  }
}

Worked Post-Mortem — Case #042

A walkthrough showing how the flow uncovers a shared root cause.

IncidentINC-2026-042 · AI agent „HelpDesk v3" · P2 · Duration 3h 20min
SymptomCost +680% in 14h; 7 users report answers referencing HR data instead of the HelpDesk knowledge base.
Impact~$1,450 extra spend; 3 users briefly shown wrong content; no PII leaked (verified via trace).
Root CauseNew RAG connector in v3.2.0 pointed to wrong workspace (HR instead of HelpDesk) via public endpoint. Service principal had Storage Blob Data Contributor at subscription scope, so access succeeded. Large HR context blew prompt-tokens P95 from 3.5k to 28k.
Detection GapBudget alert fired 6h late (threshold on cost/day only, not cost/hour); alert routed to a stale email list.
Fix (first hour)Cost-cap on → agent throttled; HR connector disabled; rolled back v3.2.0 to v3.1.7; principal scoped down to Blob-Reader + resource scope.
Lasting FixRAG source allow-list enforced in CI gate; Private Endpoint mandatory for HR storage; alerting on cost/hour & tokens/req P95; routed to PagerDuty.
Lessons(1) RBAC scope is a cost driver. (2) Daily thresholds are too slow. (3) Data-source changes need review gates like code changes.

Decision Tree (Short Form)

Cost↑ & wrong data at the same time?
├── yes → suspect shared root cause (Step 1)
│   ├── one agent_version dominates? → diff deployment (2)
│   ├── key unexplained / leaked?    → rotate, IR (3,9)
│   ├── prompt-tokens P95↑?          → retrieval/loop fix (4)
│   ├── principal over-privileged?   → tighten RBAC (5)
│   └── egress outside perimeter?    → network fix (6)
└── no → open separate incidents

Quick-Fixes (first hour)

  1. Set cost-cap — hard token & cost limit per API key/agent (circuit breaker).
  2. Throttle rate-limit — limit agent throughput until cause is clear.
  3. Toggle wrong-data source — disable affected connector / enforce allow-list.
  4. Rotate key — at any hint of a leak; low effort, high impact.
  5. Stakeholder brief — one-liner: „Cost anomaly + data-access issue, investigation underway, cost-cap active."
