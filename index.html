AI-Cloud Cost & Access Triage

Mini-Portfolio-Stück · einseitiger Triage-Runbook für interne AI-Agenten
Szenario

Ein Team nutzt einen internen AI-Agenten. Die Kosten steigen plötzlich und spürbar. Gleichzeitig greifen Antworten auf falsche / nicht autorisierte Datenquellen zu. Beide Symptome zeitgleich = typischerweise eine gemeinsame Ursache (z. B. fehlerhafter Datenquellen-Anschluss, falscher API-Key, leakender Prompt-Kontext, ausrollende RBAC-Änderung).

Architektur & Kontrollpunkte

Wo im Agent-Pfad Cost-Cap, RBAC und Private Endpoint sitzen — und wo die „falschen Daten" einsickern.

Agent-Pfad mit Kontrollpunkten. Der rote, gestrichelte Pfad (RAG → Datenquelle B) ist der Verdachtsschwerpunkt.

Die 9 Schritte im Detail

1

Symptom — Beobachtung quantifizieren

Was prüfen

  • Kosten-Delta: Tages-/Stundenschnitt vs. 7-Tage-Baseline (absolut und pro Aufruf).
  • Antwortqualität: Welche Antworten greifen auf falsche Daten zu? Reproduzierbar?
  • Zeitkorrelation: Wann begannen beide Symptome? Gleicher Timestamp = gemeinsame Ursache.

Konkrete Aktion

  • Metriken-Plot 14 Tage: cost/day, tokens/req, error_rate, Mismatch-Tickets.
  • Beispiel-Queries mit Bezug auf falsche Datenquelle fixieren (Golden-Case).
Stopp, falls: Kosten- und Qualitätssymptom zum selben Timestamp starten → als ein Incident behandeln.
2

User/Team — Betroffenheit & Verursacher

Was prüfen

  • Welche Teams/User melden falsche Antworten? Konzentriert oder verteilt?
  • Welcher Agent-Service / welche agent_version produziert die Kosten?
  • Letzte Änderungen: Feature-Rollout, neuer Datenkonnektor, Prompt-Änderung?

Konkrete Aktion

  • Aufrufe nach tenant/team/user_id und agent_version gruppieren.
  • Diff des letzten Deployments (Prompt-Template, Tool-/Function-Definitionen, Retrieval-Quellen).
Stopp, falls: ein Service-Account oder eine agent_version dominiert → dort eingrenzen.
3

Modell/API-Key — Welches Modell, welcher Key

Was prüfen

  • Teureres Modell deployt? (gpt-4o vs. 4o-mini, claude-opus vs. sonnet).
  • API-Key-Herkunft: produktiv vs. Test-/Shared-Key? Rotationsdatum?
  • Routing: Fallback auf teureres Modell bei Fehler? RAG aktiviert?

Konkrete Aktion

  • Pro-Key: requests, tokens_in/out, cost der letzten 24 h aus Billing-/Usage-API.
  • Key-Usage auf Anomalien prüfen (ungeklärte Nutzung = möglicher Leak).
Stopp, falls: Key im Code/Logs/öffentlichen Repos auftaucht → sofort rotieren, als möglichen Missbrauch behandeln.
4

Tokenkosten — Wo entstehen die Tokens

Was prüfen

  • Prompt-Tokens vs. Completion-Tokens. Hohe Prompt-Tokens = Kontext-/Retrieval-Blowup.
  • System-Prompt-Größe pro Request; mitgelieferte Doku-/Tool-Definitionen.
  • Schleifen/Retries: ruft der Agent das Modell mehrfach pro Task auf?

Konkrete Aktion

  • Verteilung prompt_tokens pro Request (P50/P95/P99). P95 » P50 = Ausreißer.
  • Top-10 Requests nach Token-Verbrauch inspizieren → oft 1–2 Verursacher.
  • Prompt-Caching für statische System-Prompts prüfen.
Stopp, falls: Prompt-Tokens P95 plötzlich 5–10× höher → verdächtig auf Datenquellen-Kontext / Schleife.
5

IAM/RBAC — Dürfen die Zugriffe sein

Was prüfen

  • Identität des Agenten: Managed Identity / Service Principal / User?
  • Verfügte vs. benötigte Berechtigungen (least privilege): Blob-Reader vs. Contributor.
  • Allow-List der Tools/Datenquellen vorhanden?

Konkrete Aktion

  • Effective-Permissions des Principals pro Datenquelle auslesen.
  • Letzte RBAC-Änderungen an Principal / Datenquellen (Audit-Log).
  • Zugriff auf Quellen außerhalb des beabsichtigten Scope? → erklärt „falsche Daten".
Stopp, falls: Principal ist Contributor/Owner auf breitem Scope → Over-Privilegierung als Ursache.
6

Private Endpoint/Netzwerk — Wer darf wohin sprechen

Was prüfen

  • Spricht der Agent nur mit Allow-List-Endpunkten? (Private Endpoint, VNet/Subnet, Egress-Rules).
  • Neue Datenquelle per Public-IP / nicht-privatem Endpunkt angebunden?
  • DNS: löst der Name auf Private Endpoint oder öffentlich auf?

Konkrete Aktion

  • Egress-/NSG-Flow-Logs 24 h prüfen → unerwartete Ziel-IPs/Domains.
  • Jede genutzte Quelle über Private/Service Endpoint verifizieren.
  • RAG-Connector mit Public-Endpoint? Daten könnten von anderem Tenant/Workspace stammen.
Stopp, falls: Agent erreicht Quellen außerhalb des Netzwerk-Perimeters → Ursache für „falsche Daten" + unkontrollierte Abrufe.
7

Logs/Traces — Was passierte wirklich

Was prüfen

  • End-to-End-Trace: User → Agent → Retrieval → Datenquelle → Modell → Antwort.
  • Wurden mehrere Quellen abgefragt? Was landete im Prompt-Kontext?
  • Tool-/Function-Call-Logs: welche Tools, welche Argumente?

Konkrete Aktion

  • Distributed-Tracing (Trace-ID) für Beispiel-Fälle ziehen.
  • Prompt-Kontext einer falschen Antwort rekonstruieren → injizierte Quellen.
  • Correlation-ID durch alle Hops; fehlende Spans = Observability-Lücken.
Stopp, falls: Trace zeigt Retrieval aus nicht-erwarteter Quelle → beweist Ursache; Fix in Schritt 5/6.
8

Budget-Alert — Greift die Frühwarnung

Was prüfen

  • Gab es einen Budget-/Anomalie-Alert? Wann hat er gefeuert?
  • Wurde er ignoriert / nicht geroutet (E-Mail-Graben, Paging fehlt)?
  • Schwellen realistisch? Anomalie-Erkennung vs. fixer Threshold.

Konkrete Aktion

  • Alert-Historie 14 Tage; Latenz zwischen Ereignis und Alert.
  • Alerting auf tokens/req-P95 und cost/hour ergänzen.
  • Ownership klären; Runbook-Link im Alert ergänzen.
Stopp, falls: Alert gefeuert, aber 6+ h unbearbeitet → Post-Incident auch Alerting-/On-Call-Prozess prüfen.
9

Eskalation — Wer muss ran

Was prüfen

  • Security-Verdacht? (geleakter Key, fremde Datenquelle, unautorisierte Egress).
  • Kostenschaden bisher (für Stakeholder-Kommunikation).
  • Zuständig: Platform, Security, Daten-Owner, Product-Owner?

Konkrete Aktion

  • Security-Verdacht → sofort IR, Key rotieren, Principal temp. sperren.
  • Pfad: L1 (Platform/Dev) → L2 (Security) → L3 (Data Owner + Management).
  • Stakeholder-Update: Symptom, Impact, Hypothese, nächste Aktion, ETA.
Stopp, falls: Security-Indizien (ungeklärte Key-Nutzung, Cross-Tenant) → nicht selbst debuggen, IR-Owner übergeben, Beweise sichern.

Cloud-CLI & Query-Snippets

Copy-Paste-Befehle je Cloud. Rohdateien unter snippets/ im Repo.

# Kosten & Usage pro API-Key / Modell (Azure Cost Management)
az consumption usage list --top 50 \
  --query "[?contains(instanceName,'ai-agent')].{name:instanceName, cost:pretaxCost, date:usageStart}"

# Effective permissions des Agent-Principals auf eine Datenquelle
az role assignment list --assignee <agent-object-id> --all \
  --query "[].{role:roleDefinitionName, scope:scope}" -o table

# Letzte RBAC-Änderungen (Audit-Log)
az monitor activity-log list --caller <agent-object-id> \
  --status Succeeded --resource-type "Microsoft.Authorization/roleAssignments"

# Private Endpoint-Verbindungen eines Storage-Account prüfen
az storage account show --name <storage> --query "privateEndpointConnections"

# Cost Explorer: tägliche Kosten pro Service/Tag (AWS)
aws ce get-cost-and-usage \
  --time-period Start=2026-06-01,End=2026-07-02 \
  --granularity DAILY --metrics "UnblendedCost" \
  --group-by Type=TAG,Key=agent_version Type=SERVICE

# CloudWatch Logs Insights: Token-Verteilung pro Request
aws logs start-query --log-group-name /ai/agent \
  --start-time $(date -d '7 days ago' +%s) --end-time $(date +%s) \
  --query-string 'filter event="llm_call"
  | stats pct(prompt_tokens,95) as p95, pct(prompt_tokens,99) as p99 by bin(1h)'

# IAM: Policy des Agent-Service-Roles auflisten
aws iam list-attached-role-policies --role-name AgentServiceRole
aws iam get-role --role-name AgentServiceRole --query "Role.MaxSessionDuration"

# GCP Billing: tägliche Kosten pro Label (agent_version)
gcloud billing accounts list
gcloud alpha billing accounts describe <ACCOUNT> --format="value(displayName)"

# BigQuery: Token-Kosten-Export pro Request aggregieren
bq query --use_legacy_sql=false '
  SELECT agent_version, APPROX_QUANTILES(prompt_tokens, 100)[OFFSET(95)] AS p95,
         SUM(cost) AS cost
  FROM `proj.ai.token_usage`
  WHERE _PARTITIONDATE >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 7 DAY)
  GROUP BY agent_version ORDER BY cost DESC'

# IAM effective permissions (Policy Analyzer)
gcloud asset analyze-iam-policy --full-resource-name=<RESOURCE> \
  --permissions="storage.objects.get" --format=json

// AppInsights/Log Analytics: Token-Verteilung + Top-Verbraucher
let span = 24h;
AIAgentLLMCall
| where timestamp > ago(span)
| summarize req=count(), p50=percentile(prompt_tokens,50),
            p95=percentile(prompt_tokens,95), p99=percentile(prompt_tokens,99),
            cost=sum(todouble(cost)) by agent_version, dataSource
| where p95 > p50 * 3   // nur Requests mit Blowup
| order by cost desc

// Trace-Hops pro Trace-ID (Observability-Lücken finden)
AppTraces
| where operation_Id == "<trace-id>"
| project timestamp, name, severityLevel, message
| order by timestamp asc

Budget-Alert & Cost-Cap as Code

Budget + Anomalie-Alert + Action-Group als IaC. Dateien: infra/budget.tf (Terraform) & infra/budget.bicep.

# infra/budget.tf — Budget + Anomalie-Alert + Action-Group (Azure)
resource "azurerm_resource_group" "ai" {
  name     = "rg-ai-agent"
  location = var.location
}

resource "azurerm_monitor_action_group" "oncall" {
  name                = "ag-ai-oncall"
  resource_group_name = azurerm_resource_group.ai.name
  short_name          = "aiOnCall"
  email_receiver {
    name          = "platform"
    email_address = "platform-oncall@example.com"
  }
}

resource "azurerm_consumption_budget_resource_group" "agent" {
  name              = "budget-ai-agent"
  resource_group_id = azurerm_resource_group.ai.id
  amount                = 500                       # EUR / Monat
  time_grain            = "Monthly"
  time_period {
    start_date = "2026-07-01T00:00:00Z"
    end_date   = "2027-07-01T00:00:00Z"
  }
  filter {
    tag { name = "agent_version", values = ["prod"] }
  }
  notification {
    enabled   = true
    threshold = 80                       # % vom Budget
    operator  = "GreaterThan"
    contact_groups = [azurerm_monitor_action_group.oncall.id]
  }
  notification {
    enabled   = true
    threshold = 100
    operator  = "GreaterThan"
    contact_groups = [azurerm_monitor_action_group.oncall.id]
  }
}

# Anomalie-Erkennung (Scheduled-Query-Alert auf cost/hour)
resource "azurerm_monitor_scheduled_query_rules_alert" "cost_spike" {
  name                = "alert-ai-cost-spike"
  resource_group_name = azurerm_resource_group.ai.name
  location            = azurerm_resource_group.ai.location
  frequency           = 5
  time_window         = 15
  data_source_id      = var.law_id
  query               = <<QUERY
AIAgentLLMCall | summarize cost = sum(todouble(cost)) by bin(1h)
| where cost > 20   # EUR/h Schwellwert
QUERY
  trigger { operator = "GreaterThan", threshold = 0 }
  action { action_groups = [azurerm_monitor_action_group.oncall.id] }
}

// infra/budget.bicep — Budget + Alert (Azure)
param location string = resourceGroup().location
param budgetAmount int = 500
param oncallEmail string = 'platform-oncall@example.com'

resource actionGroup 'Microsoft.Insights/actionGroups@2023-01-01' = {
  name: 'ag-ai-oncall'
  location: 'global'
  properties: {
    groupShortName: 'aiOnCall'
    enabled: true
    emailReceivers: [{ name: 'platform', emailAddress: oncallEmail, useCommonAlertSchema: true }]
  }
}

resource budget 'Microsoft.Consumption/budgets@2023-04-01-preview' = {
  name: 'budget-ai-agent'
  properties: {
    amount: budgetAmount
    timeGrain: 'Monthly'
    timePeriod: { startDate: '2026-07-01T00:00:00Z', endDate: '2027-07-01T00:00:00Z' }
    filter: { tags: { name: 'agent_version', values: ['prod'] } }
    notifications: {
      warn80:  { enabled: true, operator: 'GreaterThan', threshold: 80,  contactEmails: [oncallEmail] }
      over100: { enabled: true, operator: 'GreaterThan', threshold: 100, contactEmails: [oncallEmail] }
    }
  }
}

Prevention & Guardrails

Reaktiv ist zu spät — diese Guardrails hätten beide Post-Mortems verhindert oder eingedämmt.

  • CI-Gate für RAG-Quellen-Allow-List — neue Datenquellen brauchen Review wie Code-Änderungen; CI bricht bei nicht gelisteter Quelle.
  • Least-Privilege per IaC — Principals nur Resource-Scope + Blob-Reader; Contributor/Owner auf Subscription sind verboten (Policy).
  • Prompt-Size-Limit im Code — harte Obergrenze für prompt_tokens; Überschreitung bricht Request + Alert statt eines stillen Blowups.
  • Cost-Cap & Rate-Limit pro Key — Circuit-Breaker im Agent-Layer, nicht nur Cloud-seitig (siehe infra/).
  • Recall-/Eval-Gate für Modellwechsel — Embedding-/Modell-Änderungen brauchen Eval-Suite (Recall, Mismatch-Rate) in CI.
  • Alerting auf P95 & cost/hour — nicht nur cost/day; Routing an PagerDuty, nicht nur E-Mail-Liste.
  • Private Endpoint erzwungen — Azure Policy / Org Policy verbietet Public-Endpoint für Datenquellen.
  • Trace-Coverage-SLO — trace-completeness > 0,9 als SLO; Lücken werden selbst zum Ticket.

Worked Post-Mortems

Zwei durchgespielte Fälle mit unterschiedlichen Root Causes — derselbe Triage-Fluss deckt beide auf.

IncidentINC-2026-042 · AI-Agent „HelpDesk v3" · P2 · Dauer 3h 20min
SymptomKosten +680 % in 14 h; 7 User melden Antworten mit Verweis auf HR-Daten statt auf HelpDesk-Wissensbasis.
Impact~1.450 EUR Mehraufwand; 3 User vorübergehend falsche Inhalte; keine PII geleakt (via Trace verifiziert).
Root CauseNeuer RAG-Connector in v3.2.0 auf falschen Workspace (HR statt HelpDesk) konfiguriert — via Public-Endpoint. Service-Principal hatte Storage Blob Data Contributor auf Subscription-Scope. Großer HR-Kontext blies Prompt-Tokens P95 von 3,5k auf 28k.
Detection-GapBudget-Alert feuerte 6 h spät (nur cost/day); Alert ging an inaktive E-Mail-Liste.
Fix (1. Stunde)Cost-Cap an; HR-Connector aus; v3.2.0 → Rollback v3.1.7; Principal auf Blob-Reader + Resource-Scope.
Dauer-FixRAG-Allow-List als CI-Gate; Private Endpoint für HR-Storage erzwungen; Alerting auf cost/hour & tokens/req P95; PagerDuty.
Lessons(1) RBAC-Scope ist eine Kostenursache. (2) Tages-Thresholds sind zu träge. (3) Datenquellen brauchen Review-Gate wie Code.

IncidentINC-2026-051 · AI-Agent „HelpDesk v3" · P2 · Dauer 5h 40min
SymptomKosten +210 % über 3 Tage; Antworten zitieren veraltete/falsche Dokumente aus den korrekten (erlaubten) Quellen.
Impact~2.300 EUR Mehraufwand; 12 User mit suboptimalen Antworten; keine Datenzugriffs-Verletzung (Quellen erlaubt, Inhalte nur falsch).
Root CauseEmbedding-Modell für RAG ohne Recall-Eval getauscht. Recall sank; eine „Top-K-Inflation"-Heuristik holte 8× mehr Chunks als Ausgleich → Prompt-Tokens P95 von 3,5k auf 26k, und falsche Chunks landeten im Kontext.
Detection-GapKein Recall-/Mismatch-Regressionstest in CI; Alert nur auf cost/day, nicht auf tokens/req P95.
Fix (1. Stunde)Embedding-Modell zurückgerollt; Top-K gedeckelt auf 5; Cost-Cap aktiv; Antworten-Fehlerquote sank sofort.
Dauer-FixEval-Suite (Recall + Mismatch-Rate) als CI-Gate für Modellwechsel; harte prompt_tokens-Obergrenze im Code; Alert auf P95.
Lessons(1) Modelländerungen brauchen Eval-Gates. (2) Kompensations-Heuristiken können Kosten amplifizieren. (3) „Falsche Daten" muss nicht „falsche Quelle" heißen — Recall ist eine Kostenursache.

Quick-Fixes (erste Stunde)

  1. Cost-Cap setzen — pro API-Key/Agent harte Token- & Kostenobergrenze (Circuit-Breaker).
  2. Rate-Limit drosseln — bis Ursache klar ist, Agent-Durchsatz begrenzen.
  3. Falsche-Daten-Source togglen — betroffenen Konnektor deaktivieren / auf Allow-List setzen.
  4. Key rotieren — bei minimalem Leak-Verdacht; Aufwand gering, Wirkung hoch.
  5. Stakeholder-Briefing — 1-Zeiler: „Kostenanomalie + Datenzugriffs-Problem, Untersuchung läuft, Cost-Cap aktiv."
