AI-Cloud Cost & Access Triage

Mini-Portfolio-Stück · einseitiger Triage-Runbook für interne AI-Agenten
Szenario

Ein Team nutzt einen internen AI-Agenten. Die Kosten steigen plötzlich und spürbar. Gleichzeitig greifen Antworten auf falsche / nicht autorisierte Datenquellen zu. Beide Symptome zeitgleich = typischerweise eine gemeinsame Ursache (z. B. fehlerhafter Datenquellen-Anschluss, falscher API-Key, leakender Prompt-Kontext, ausrollende RBAC-Änderung).

1 Symptom→ 2 User/Team→ 3 Modell/API-Key→ 4 Tokenkosten→ 5 IAM/RBAC→ 6 Private Endpoint/Netzwerk→ 7 Logs/Traces→ 8 Budget-Alert→ 9 Eskalation
1

Symptom — Beobachtung quantifizieren

Was prüfen

  • Kosten-Delta: Tages-/Stundenschnitt vs. 7-Tage-Baseline (absolut und pro Aufruf).
  • Antwortqualität: Welche Antworten greifen auf falsche Daten zu? Reproduzierbar?
  • Zeitkorrelation: Wann begannen beide Symptome? Gleicher Timestamp = gemeinsame Ursache.

Konkrete Aktion

  • Metriken-Plot 14 Tage: cost/day, tokens/req, error_rate, Mismatch-Tickets.
  • Beispiel-Queries mit Bezug auf falsche Datenquelle fixieren (Golden-Case).
Stopp, falls: Kosten- und Qualitätssymptom zum selben Timestamp starten → als ein Incident behandeln.
2

User/Team — Betroffenheit & Verursacher

Was prüfen

  • Welche Teams/User melden falsche Antworten? Konzentriert oder verteilt?
  • Welcher Agent-Service / welche agent_version produziert die Kosten?
  • Letzte Änderungen: Feature-Rollout, neuer Datenkonnektor, Prompt-Änderung?

Konkrete Aktion

  • Aufrufe nach tenant/team/user_id und agent_version gruppieren.
  • Diff des letzten Deployments (Prompt-Template, Tool-/Function-Definitionen, Retrieval-Quellen).
Stopp, falls: ein Service-Account oder eine agent_version dominiert → dort eingrenzen.
3

Modell/API-Key — Welches Modell, welcher Key

Was prüfen

  • Teureres Modell deployt? (gpt-4o vs. 4o-mini, claude-opus vs. sonnet).
  • API-Key-Herkunft: produktiv vs. Test-/Shared-Key? Rotationsdatum?
  • Routing: Fallback auf teureres Modell bei Fehler? RAG aktiviert?

Konkrete Aktion

  • Pro-Key: requests, tokens_in/out, cost der letzten 24 h aus Billing-/Usage-API.
  • Key-Usage auf Anomalien prüfen (ungeklärte Nutzung = möglicher Leak).
Stopp, falls: Key im Code/Logs/öffentlichen Repos auftaucht → sofort rotieren, als möglichen Missbrauch behandeln.
4

Tokenkosten — Wo entstehen die Tokens

Was prüfen

  • Prompt-Tokens vs. Completion-Tokens. Hohe Prompt-Tokens = Kontext-/Retrieval-Blowup.
  • System-Prompt-Größe pro Request; mitgelieferte Doku-/Tool-Definitionen.
  • Schleifen/Retries: ruft der Agent das Modell mehrfach pro Task auf?

Konkrete Aktion

  • Verteilung prompt_tokens pro Request (P50/P95/P99). P95 » P50 = Ausreißer.
  • Top-10 Requests nach Token-Verbrauch inspizieren → oft 1–2 Verursacher.
  • Prompt-Caching für statische System-Prompts prüfen.
Stopp, falls: Prompt-Tokens P95 plötzlich 5–10× höher → verdächtig auf Datenquellen-Kontext / Schleife.
5

IAM/RBAC — Dürfen die Zugriffe sein

Was prüfen

  • Identität des Agenten: Managed Identity / Service Principal / User?
  • Verfügte vs. benötigte Berechtigungen (least privilege): Blob-Reader vs. Contributor.
  • Allow-List der Tools/Datenquellen vorhanden?

Konkrete Aktion

  • Effective-Permissions des Principals pro Datenquelle auslesen.
  • Letzte RBAC-Änderungen an Principal / Datenquellen (Audit-Log).
  • Zugriff auf Quellen außerhalb des beabsichtigten Scope? → erklärt "falsche Daten".
Stopp, falls: Principal ist Contributor/Owner auf breitem Scope → Over-Privilegierung als Ursache.
6

Private Endpoint/Netzwerk — Wer darf wohin sprechen

Was prüfen

  • Spricht der Agent nur mit Allow-List-Endpunkten? (Private Endpoint, VNet/Subnet, Egress-Rules).
  • Neue Datenquelle per Public-IP / nicht-privatem Endpunkt angebunden?
  • DNS: löst der Name auf Private Endpoint oder öffentlich auf?

Konkrete Aktion

  • Egress-/NSG-Flow-Logs 24 h prüfen → unerwartete Ziel-IPs/Domains.
  • Jede genutzte Quelle über Private/Service Endpoint verifizieren.
  • RAG-Connector mit Public-Endpoint? Daten könnten von anderem Tenant/Workspace stammen.
Stopp, falls: Agent erreicht Quellen außerhalb des Netzwerk-Perimeters → Ursache für "falsche Daten" + unkontrollierte Abrufe.
7

Logs/Traces — Was passierte wirklich

Was prüfen

  • End-to-End-Trace: User → Agent → Retrieval → Datenquelle → Modell → Antwort.
  • Wurden mehrere Quellen abgefragt? Was landete im Prompt-Kontext?
  • Tool-/Function-Call-Logs: welche Tools, welche Argumente?

Konkrete Aktion

  • Distributed-Tracing (Trace-ID) für Beispiel-Fälle ziehen.
  • Prompt-Kontext einer falschen Antwort rekonstruieren → injizierte Quellen.
  • Correlation-ID durch alle Hops; fehlende Spans = Observability-Lücken.
Stopp, falls: Trace zeigt Retrieval aus nicht-erwarteter Quelle → beweist Ursache; Fix in Schritt 5/6.
8

Budget-Alert — Greift die Frühwarnung

Was prüfen

  • Gab es einen Budget-/Anomalie-Alert? Wann hat er gefeuert?
  • Wurde er ignoriert / nicht geroutet (E-Mail-Graben, Paging fehlt)?
  • Schwellen realistisch? Anomalie-Erkennung vs. fixer Threshold.

Konkrete Aktion

  • Alert-Historie 14 Tage; Latenz zwischen Ereignis und Alert.
  • Alerting auf tokens/req-P95 und cost/hour ergänzen.
  • Ownership klären; Runbook-Link im Alert ergänzen.
Stopp, falls: Alert gefeuert, aber 6+ h unbearbeitet → Post-Incident auch Alerting-/On-Call-Prozess prüfen.
9

Eskalation — Wer muss ran

Was prüfen

  • Security-Verdacht? (geleakter Key, fremde Datenquelle, unautorisierte Egress).
  • Kostenschaden bisher (für Stakeholder-Kommunikation).
  • Zuständig: Platform, Security, Daten-Owner, Product-Owner?

Konkrete Aktion

  • Security-Verdacht → sofort IR, Key rotieren, Principal temp. sperren.
  • Pfad: L1 (Platform/Dev) → L2 (Security) → L3 (Data Owner + Management).
  • Stakeholder-Update: Symptom, Impact, Hypothese, nächste Aktion, ETA.
Stopp, falls: Security-Indizien (ungeklärte Key-Nutzung, Cross-Tenant) → nicht selbst debuggen, IR-Owner übergeben, Beweise sichern.

Entscheidungsbaum (Kurzform)

Kosten↑ & falsche Daten gleichzeitig?
├── ja → gemeinsame Ursache vermuten (Schritt 1)
│   ├── eine agent_version dominiert? → Diff Deployment (2)
│   ├── Key ungeklärt / leaked?       → rotieren, IR (3,9)
│   ├── Prompt-Tokens P95↑?           → Retrieval-/Schleifen-Fix (4)
│   ├── Principal over-privileged?    → RBAC eindämmen (5)
│   └── egress außerhalb Perimeter?   → Netzwerk-Fix (6)
└── nein → getrennte Incidents öffnen

Quick-Fixes (erste Stunde)

  1. Cost-Cap setzen — pro API-Key/Agent harte Token- & Kostenobergrenze (Circuit-Breaker).
  2. Rate-Limit drosseln — bis Ursache klar ist, Agent-Durchsatz begrenzen.
  3. Falsche-Daten-Source togglen — betroffenen Konnektor deaktivieren / auf Allow-List setzen.
  4. Key rotieren — bei minimalem Leak-Verdacht; Aufwand gering, Wirkung hoch.
  5. Stakeholder-Briefing — 1-Zeiler: "Kostenanomalie + Datenzugriffs-Problem, Untersuchung läuft, Cost-Cap aktiv."
